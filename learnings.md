# Learnings

- 2026-03-24: Root `pnpm test` should use `pnpm -r --if-present test` during staged rollout so packages without a test script (like `packages/web` before T02) do not fail the workspace run. Vitest package configs can use `passWithNoTests: true` to keep the suite green while infrastructure lands before actual test files.
- 2026-03-24: For React + Vitest in this repo, use `import '@testing-library/jest-dom/vitest'` in a dedicated setup file (e.g. `src/test/setup.ts`) and register it via `setupFiles` in `vitest.config.ts`; this enables jest-dom matchers without needing Jest.
- 2026-03-24: In Playwright `webServer.command`, pass script args through pnpm without an extra `--` when filtering a workspace package (e.g. `pnpm --filter @blather/web dev --host 127.0.0.1 --port 8080`). Using `dev -- --host ...` leaves Vite on its default port (5173), causing `webServer` health checks for `localhost:8080` to time out.
- 2026-03-24: For `packages/web` Vitest setup, explicitly include TSX tests (`include: ['src/**/*.test.{ts,tsx}']`) with `environment: 'jsdom'`; otherwise component test files are skipped or fail due to missing DOM APIs.
- 2026-03-24: In ESM test helpers, resolve Drizzle `migrationsFolder` from `import.meta.url` (e.g. `fileURLToPath(new URL('../../../db/drizzle', import.meta.url))`) rather than relying on cwd; Vitest and package scripts can run with different working directories.
- 2026-03-24: For API route unit tests, export a `createApp(db)` factory instead of only a singleton app so tests can inject an isolated test DB handle; keep `export const app = createApp()` for production entrypoints.
- 2026-03-24: In this repo, API harness tests that touch Postgres should be gated behind `TEST_DATABASE_URL` (e.g. `describe.skip` when unset) so default `pnpm test` stays green on machines without a local `blather_test` database.
- 2026-03-24: For mixed auth test suites, keep pure helper tests (e.g. `signToken`, `hashApiKey`) outside the `TEST_DATABASE_URL`-gated describe block so CI/dev runs without Postgres still execute meaningful coverage while DB-backed middleware route checks remain conditionally skipped.
- 2026-03-24: Auth route tests can verify `/auth/magic` without mocking Resend by asserting DB side effects (`magic_tokens` row creation, lowercased email, `usedAt` transitions). This keeps coverage deterministic even when `RESEND_API_KEY` is unset and email sending is intentionally skipped in local/test environments.
- 2026-03-24: `GET /workspaces/:id/unread` only returns channels with unread messages because the SQL uses `HAVING COUNT(m.id) > 0`; tests should assert missing keys for read channels rather than expecting explicit zeroes.
- 2026-03-24: Channel-route tests can verify message pagination by seeding one thread reply and asserting `GET /channels/:id/messages` excludes it (`threadId IS NULL`), while `GET /channels/:channelId/messages/:messageId/replies` returns it; this guards both code paths with minimal fixture setup.
- 2026-03-24: `GET /messages/search` computes access as all public channels in the workspace plus only private/DM channels listed in `channel_members` for the requester. For deterministic search tests, use public channels unless membership visibility is the behavior under test.
- 2026-03-24: Postgres `numeric`/`decimal` columns (e.g. `runwayMonths`, `headcount` in `portfolio_metrics`) are returned as strings through Drizzle's JSON serialization. Use `Number(value)` in test assertions rather than direct `.toBe(18)`.
- 2026-03-24: When mocking `fs` at the module level with `vi.mock('fs', ...)`, `vi.clearAllMocks()` in `beforeEach` does NOT reset `mockImplementation` — only calls/results. Tests that set `mockImplementation` (e.g. for `existsSync` cache hits) will leak into subsequent tests. Fix: explicitly re-set the default implementation in `beforeEach`.
- 2026-03-28: For module-level in-memory stores with timers, load a fresh module per test (`vi.resetModules()` + dynamic `import()`) and use fake timers. Otherwise singleton state and pending autoclear callbacks leak between cases and make timer assertions flaky.
- 2026-03-24: Migration 0006 (`0006_dizzy_cassandra_nova.sql`) was a Drizzle-generated catch-up migration that duplicated objects from 0004 (`attachments` column) and 0005 (`portfolio_metrics` table + indexes). Fixed by removing the duplicate `ADD COLUMN` and adding `IF NOT EXISTS` to the `CREATE TABLE` and `CREATE INDEX` statements. When Drizzle generates a migration after hand-written ones, always diff for overlap.
- 2026-03-24: For WS manager tests, mock `@blather/db` at module level (the manager creates `db` at import time via `createDb()`). Use a queue-indexed mock where each `db.select()` chain resolves to `dbQueryResults[dbQueryIndex++]`, reset in `beforeEach`. This avoids needing a real Postgres connection for pure WS behavior tests.
- 2026-03-24: When testing WebSocket message ordering, avoid queue-based message collectors that re-read from their own queue (busy-loop risk). Instead use `waitForType(ws, type)` which registers a filtered `ws.on('message')` listener — non-matching messages are silently ignored. Register listeners BEFORE the action that triggers the message to avoid race conditions with Node.js event loop scheduling.
- 2026-03-24: In jsdom component tests, `Element.prototype.scrollIntoView` is not implemented. Mock it with `Element.prototype.scrollIntoView = vi.fn()` in `beforeAll`. Also add `afterEach(() => cleanup())` when tests accumulate DOM nodes across renders.
- 2026-03-24: jsdom normalizes inline CSS hex colors to `rgb()` format (e.g. `#3366CC` → `rgb(51, 102, 204)`). Assert on the rgb form, not hex. Similarly, `style.border` returns the full shorthand like `1px solid rgb(...)`.
- 2026-03-24: When SearchPanel's `highlight()` splits text by regex match, `findByText(/pattern/)` won't find the full content because it is split across `<span>` elements. Search for a non-highlighted field (like `userName`) or use `{ exact: false }`.
- 2026-03-24: For E2E tests using Playwright's dev magic link flow: the API returns `_dev.token` when `RESEND_API_KEY` is unset. Use `registerUser` + `loginWithToken` (inject JWT into localStorage) for fast setup, or `loginViaMagicLink` for the full UI flow. Each test should use unique emails with timestamps to avoid conflicts.
- 2026-03-26: For `AuthPage` UI tests, reset both mock state and `window.history` in `beforeEach`. The component mutates browser history during sign-in, so carrying URL state across tests can create confusing cross-test coupling; role-based button queries (`getByRole`) also read more cleanly than raw text selectors for the send/verify actions.
- 2026-03-26: When a UI component duplicates shared modal chrome, first check whether it can reuse the common `Modal` wrapper. That simplification also makes tests easier to write with accessible selectors (`getByLabelText`, `findByRole('alert')`) instead of brittle text-node lookups.
- 2026-03-26: `packages/web/src/components/Modals.test.tsx` is the shared test file for several modal components. When reviewing one modal, extend the relevant describe block there instead of creating another tiny test file; assert on the API payload (`name`, generated `slug`, normalized arrays) and accessible error state so tests stay behavioral even if helpers are refactored.
- 2026-03-26: For modal form tests in `packages/web`, wire labels to inputs with `htmlFor`/`id` and prefer `getByLabelText(...)` over placeholder queries. Also add an explicit whitespace-only validation test when the component trims input, because browser `required` alone does not cover that behavior.
- 2026-03-26: For channel-creation UI, validate both the trimmed name and the derived slug before calling the API. A value like `!!!` satisfies the browser's `required` constraint but normalizes to an empty slug, so tests should assert the modal surfaces a validation alert and skips the network call.
- 2026-03-26: Context-menu components are much easier to test and reason about when menu actions are real `button`s instead of clickable `div`s. That gives stable `getByRole('button', { name })` selectors, built-in disabled behavior for non-actionable items like a default channel's archive action, and removes hover-style mutation noise from the logic under test.
- 2026-03-26: When a test bundles two independent behaviors (e.g. delete action + invite action) with intermediate `cleanup()`/re-render, split them into separate tests. This satisfies Kent Beck's "specific" desideratum: a failure points precisely at the broken behavior rather than masking the second assertion behind an earlier failure. Also, for non-accessible elements like overlay backdrops (`aria-hidden`), prefer `data-testid` over DOM-traversal selectors like `.previousElementSibling` — the latter breaks if sibling order changes and couples the test to layout structure.
- 2026-03-26: When a component uses raw `fetch` with manual token/auth instead of the shared `api` module, first add the missing method to `api.ts` and refactor the component to use it. This keeps tests mockable via `vi.mock('../lib/api')` (consistent with all other modal tests) and removes duplicated auth/error-handling logic. Also add the mock entry to the shared mock object in `Modals.test.tsx` when extending the suite.
- 2026-03-26: In large `onWsEvent`-style callback functions with many `if (event.type === '...')` branches, watch for copy-paste corruption where blocks from one event handler get duplicated inside another handler's braces. In `MainPage.tsx`, the `member.joined` handler had 8 duplicate copies of `message.updated`/`message.deleted` interleaved within it — dead code that couldn't execute (since `event.type` can only be one value) but bloated the file by ~56 lines and obscured the actual logic. A `vi.stubGlobal('fetch', ...)` in the test was a smell pointing to the component bypassing the `api` module; removing the raw fetch and using `api.getActiveHuddles` let us drop the global stub entirely.
- 2026-03-26: When a component uses `setState(prev => { /* read prev, return prev unchanged */ })` just to peek at current state inside a callback, that's a misuse of the setter — it triggers a re-render for no reason and confuses future readers. Instead, mirror the state into a `useRef` (`transcriptData.current = transcript` in a sync effect) and read the ref from callbacks. Applied in `HuddleModal.tsx` where `playNext` was using `setTranscript` to find the speaking user.
- 2026-03-26: For `HuddleModal` tests in jsdom, stub `Audio` and `AudioContext` globally with `vi.stubGlobal` in `beforeAll`, and mock `Element.prototype.scrollIntoView`. Import `beforeAll` from vitest — the existing `Modals.test.tsx` only had `beforeEach` imported. Use `rerender()` with new `huddleEvents` arrays to test event processing, since the component diffs new events via a `lastProcessedIndex` ref.
- 2026-03-28: For API route tests that import `app.ts`, mock `../ws/manager.js` before importing the harness when you only need to assert status/publish side effects. That avoids loading the real WebSocket manager singleton (which creates its own DB handle at import time) and makes it easy to assert broadcast payloads directly. Route tests that hit module-level in-memory stores (like `agentStatus`) also need an explicit store reset in `beforeEach`, because `harness.reset()` only clears the PGlite DB. Add an explicit whitespace-only validation case for text fields like `/status` so the test covers trimmed input behavior, not just missing keys.
- 2026-03-28: Shared modal tests are cleaner and less structure-coupled when `Modal.tsx` exposes real dialog semantics (`role="dialog"`, `aria-labelledby`) plus an actual close `button` with an accessible name. For backdrop clicks, a dedicated `data-testid="modal-overlay"` is preferable to `container.firstElementChild`, which bakes DOM shape into the assertion.
- 2026-03-28: Before polishing or testing a UI component, first confirm it is still mounted from a page entrypoint. `packages/web/src/components/MenuBar.tsx` had drifted into an orphaned duplicate of the inline menu bar in `MainPage.tsx`; deleting the dead file was cleaner than preserving unexercised placeholder alerts and menu state.
- 2026-03-28: For CLI helpers that hardcode repo-level paths like `ENV_PATH`, prefer mocking `node:fs` with a hoisted in-memory file map instead of touching real temp files or the workspace `.env`. This keeps tests isolated while still covering behaviors like cloning `.env.example`, uncommenting matching keys, and appending new settings.
- 2026-03-28: For route tests that exercise timer-driven side effects through the full Hono app (for example `/status` auto-clear broadcasting), use `vi.useFakeTimers()` inside the specific test, advance with `await vi.advanceTimersByTimeAsync(...)`, and restore real timers in `afterEach`. That keeps the test behavioral while avoiding timer leakage into the shared API harness.
- 2026-03-28: `packages/cli/src/down.ts` currently kills processes by port and with a broad `pgrep -f "tsx.*blather|node.*blather.*(api|web)"` sweep, but there are no tests covering this module yet. That makes it easy for the cleanup command to become over-broad; a focused, mock-based unit test around `killPort`/process matching would make the behavior safer to change.
- 2026-03-28: `packages/cli/src/openclaw.ts` contains a lot of direct `openclaw` / filesystem / network orchestration, but only the pure helpers (`parseChannelRefs`, `applyTemplateVars`, `AGENT_DOMAIN`) are currently covered by tests. If you revisit the command flow, consider extracting smaller pure helpers first so behavior can be tested without shelling out to the real CLI.
- 2026-03-28: `packages/cli/src/setup.ts` now has a focused test that stubs `node:fs`, `node:readline/promises`, `node:process`, and `node:crypto`; when adding more setup coverage, keep the prompt queue explicit and avoid real terminal I/O so the quick/advanced flows stay deterministic.
- 2026-03-28: `packages/db` is purely declarative (Drizzle schema definitions + a thin `createDb` factory + a migration script). Unit-testing these files would just be testing Drizzle itself; real coverage comes transitively from the API route tests that exercise the schema against PGlite. Don't create test files for pure schema packages. Also delete stale `.backup` files — they drift immediately and add confusion.
- 2026-03-28: The `openclaw_blather` plugin depends on `openclaw/plugin-sdk` provided at runtime by the OpenClaw plugin loader — it is not installed as a regular npm dependency. Unit tests for this plugin's pure-wiring entry point (`index.ts`) would require mocking the entire SDK and would only be implementation traces. Same rationale as the `packages/db` schema: test the composed modules individually where they contain real logic (config resolution, API client, etc.), not the declarative glue.
